Pitfall #1 ‚Äì Mongo cannot write data to the host OSX machine

There are three factors that cause this issue:

  1. A lot of mongo/docker docs tell you to mount a local volume
  2. By default docker-machine uses the virtualbox driver to create a local VM

These facts led me (by way of this issue) to the fact that there‚Äôs been an ancient known issue on VirtualBox that basically says docker ain‚Äôt going to be able to read/write files with mmap.

So it‚Äôs pointless trying to tell mongo, via docker, to mount a local data volume, as the above bug means mongo isn‚Äôt going to be able to access it. (careful reading of the Docker Hub mongo docs may allow you to divine this, but it‚Äôs not at all obvious)

Solution ‚Äì Create a data volume in the container rather than the host.

In my (still) early days with docker I‚Äôd come across these cli commands which contained absolutely no description of WTF all the switches are doing, so a quick breakdown:

  • ‚Äìname my-local-mongo ‚Äì This tells docker to give the container the name ‚Äúmy-local-mongo‚Äù rather than generating a name like tuftybeaver
  • -v mongo-data:/data/db ‚Äì This is the key switch, here we tell docker to create a new data volume called mongo-data and mount it as /data/db which is the default location mongo wants to write data to
  • -p 27017:27017 ‚Äì Here we tell docker to bind the container port 27017 to port 27017 on the host vm, allowing us to access docker locally.
  • -d ‚Äì This switch tells docker to run the process detached so we‚Äôre not running interactively.

If all has gone well docker should be running your mongo container successfully. A quick docker ps will show your running container.

docker logs -f my-local-mongo ‚Äì will allow you to follow the log files

This is because everything is actually running on your local docker virtual machine. To find the IP of the VM enter docker-machine ls
